# 第九章 省域绿色算力承载能力评估与资源布局决策支持系统设计与实现

## 9.1 系统建设目标与功能定位

随着国家"东数西算"工程的深入推进和"双碳"目标的加速落地，省域绿色算力承载能力已成为衡量区域数字经济发展潜力的核心指标之一。前文章节已从指标体系构建、TOPSIS综合评价、Dagum区域差异分解、Moran's I空间自相关分析、Markov动态演化模型、LPA潜在剖面分析、SHAP可解释性分析和障碍度诊断等方面，完成了从数据到模型再到结论的完整分析链条。然而，静态的论文分析和离线的模型结果在实际决策场景中存在三个核心瓶颈：一是模型结果分散在多个文件中，难以被非技术用户统一查询和理解；二是分析结论以静态图表形式呈现，缺乏交互式探索能力；三是不同用户（政府部门、数据中心企业、能源规划机构）的查询需求差异大，静态报告无法满足个性化和即时性的决策支持需求。

在此背景下，本章设计并实现了"省域绿色算力承载能力评估与资源布局决策支持系统"（以下简称"本系统"）。系统以2016—2024年中国31个省级行政区的多源面板数据为基础，将前文章节的模型分析结果工程化封装为可交互、可查询、可展示的Web数据驾驶舱，并通过RAG检索增强生成技术实现自然语言智能问答，构建了从"数据汇聚—模型评价—空间诊断—类型识别—布局决策—智能问答"的完整应用闭环。

本系统的核心建设目标包括以下四个方面：

**（1）模型结果的产品化封装。** 将分散在数据库中的TOPSIS综合得分、Dagum分解结果、Moran's I统计量、Markov转移矩阵、LPA类型分类、障碍度诊断和SHAP解释结果，通过标准化的RESTful API接口和可视化图表组件，转化为可被非技术用户直接访问和理解的数字化产品。

**（2）多维度交互式可视化。** 基于ECharts构建10个功能页面的数据驾驶舱，支持中国地图、南丁格尔玫瑰图、环形饼图、热力矩阵、折线面积图、雷达图、SHAP瀑布图、横向排名条等十余种高级可视化形式，所有图表支持鼠标悬停tooltip、图例筛选、缩放拖拽和省份切换等交互操作。

**（3）自然语言智能问答。** 集成RAG检索增强生成技术和DeepSeek大语言模型，用户可通过自然语言查询绿色算力排名、省份诊断、布局建议和趋势预测等问题，系统自动分类问题类型、检索知识库文档、调用数据库工具，并生成结构化且可追溯的回答。

**（4）完整的决策支持闭环。** 系统面向政府部门、数据中心企业、能源规划机构和研究人员四类用户，提供从全国总览、区域对比、空间分析、类型识别、省域诊断到布局决策和智能问答的全链条决策支持服务，体现数据要素从"数据资源"到"分析结果"再到"决策服务"的价值转化路径。

## 9.2 系统总体架构设计

本系统采用前后端分离的B/S（Browser/Server）架构，整体设计遵循"数据资源层—数据存储层—后端服务层—前端展示层—应用交互层"的五层架构模式。各层之间通过标准化的RESTful API接口和数据模型实现松耦合协作，确保系统的可维护性和可扩展性。

### 9.2.1 数据资源层

数据资源层是系统的数据底座，汇聚了以下四类数据资源：

（1）**原始指标数据**：31个省级行政区2016—2024年共计9个年度的面板数据，涵盖算力需求基础、能源供给能力、数字基础设施、绿色低碳约束、创新与人才支撑、气候与自然条件、区域协同能力等8个一级维度的34个二级指标，原始数据来源于国家统计局、中国能源统计年鉴、中国科技统计年鉴和中国信息产业年鉴等权威统计资料。

（2）**模型计算结果数据**：基于前文章节模型方法计算生成的全部分析结果，包括TOPSIS综合得分与排名、熵权法+CRITIC组合权重、Dagum基尼系数分解（总体、区域内、区域间）、Moran's I全局空间自相关与LISA局部聚集分析、传统Markov与空间Markov转移概率矩阵、LPA四种发展类型分类、障碍度诊断结果（省域维度障碍度、指标障碍度）、SHAP特征贡献值（全局重要性与局部解释）、布局决策推荐结果（优势指数、适宜度、功能定位、优化策略）等。

（3）**知识库文档**：为支撑智能问答模块的RAG检索增强生成，系统整理了10篇专业知识文档，涵盖系统概述、TOPSIS模型说明、LPA模型说明、空间演化分析、障碍度模型、SHAP可解释性、布局决策规则、未来预测规则、指标体系参考和常见问题解答。

（4）**地理信息数据**：中国省级行政区划GeoJSON数据，用于前端ECharts中国地图的渲染和空间可视化。

### 9.2.2 数据存储层

数据存储层以PostgreSQL关系型数据库为核心，通过SQLAlchemy ORM进行统一的数据访问管理。数据库共包含42张数据表，按业务领域分为8个类别：基础信息类（省份基本信息、邻接矩阵、指标体系定义）、核心数据类（指标原始值、标准化值、组合权重、TOPSIS得分）、LPA分析类（省份分类、类型汇总、模型拟合、轨迹数据）、空间演化类（Dagum分解、区域内基尼、区域间基尼、Moran's I、LISA、Markov转移概率、空间Markov、状态阈值、空间滞后状态、KDE统计）、障碍度诊断类（全国障碍度、区域障碍度、LPA障碍度、省域障碍度、指标障碍度、年度演化、省域指标详情）、SHAP解释类（模型性能指标、预测结果、全局重要性、年度维度贡献、LPA维度贡献、省域汇总、局部Top 8解释）、布局决策类（省域决策、类型汇总、类型特征、LPA布局矩阵、策略库、策略优先级）和知识库文档类。

数据库以"年份—省份—指标—模型结果"为主线组织数据关系。所有模型结果表通过省份名称和年份字段与基础信息表关联，支持按年度查询、按省份筛选、按区域聚合和跨模型关联查询。数据库连接采用连接池管理（pool_size=5, max_overflow=10），启用pool_pre_ping确保连接有效性。数据库初始化通过Excel数据导入管道完成：读取10个原始Excel文件，经数据清洗、字段映射、类型转换、省份/区域标准化、字典丰富和完整性校验后批量写入PostgreSQL。

### 9.2.3 后端服务层

后端服务层基于Python FastAPI框架构建，提供RESTful API数据接口和智能问答服务。FastAPI具有异步高性能、自动交互式API文档（Swagger UI和ReDoc）、Pydantic数据校验和类型安全等特性，适合构建数据密集型的Web服务后端。

后端采用分层架构设计：API路由层（`api/routes.py`）负责请求接收、参数校验和响应组装；业务服务层（`services/`目录下的11个独立服务模块）负责数据库查询、业务逻辑封装和结果转换；数据模型层（`models.py`）定义23个SQLAlchemy ORM模型映射数据库表结构；数据校验层（`schemas.py`）定义30个Pydantic响应模型确保输出数据格式的一致性。

系统共提供14个API端点：12个GET类型的数据查询接口（省份信息、综合得分、省份画像、障碍度诊断、LPA分类、SHAP解释、布局决策、首页总览、指标体系、Dagum分解、Moran's I分析、Markov转移矩阵）、1个POST类型的智能问答接口、1个健康检查接口。所有接口均采用统一的异常处理机制：`ValueError`映射为HTTP 404（数据未找到），其他异常映射为HTTP 500（内部错误），智能问答接口额外实现了永不抛出异常的安全策略。

后端服务层同时负责CORS跨域支持配置（开发阶段允许所有来源）、API文档自动生成和智能问答服务的编排（问题分类→RAG检索→数据库工具调用→LLM生成回答）。

### 9.2.4 前端展示层

前端展示层基于Next.js 15框架（App Router模式）和React 19构建，使用TypeScript进行类型安全开发，采用Tailwind CSS 3.4实现深色科技风响应式样式系统。数据可视化方面，统一使用ECharts 5.5图表库（通过echarts-for-react 3.0封装）实现全部图表的渲染和交互。

前端采用数据驾驶舱的经典布局：左侧固定宽度（224px）的导航侧边栏 + 顶部状态栏（56px） + 中央内容区（自适应滚动）。侧边栏按"总览→评价分析（4项）→诊断决策（4项）→智能（1项）"四级分组组织10个功能页面的路由导航，当前页面以蓝色高亮标识。顶部栏展示系统全称和数据就绪状态标签。

前端共实现10个功能页面：首页总览（/dashboard）、综合评价（/evaluation）、区域差异（/regional-difference）、空间聚焦（/spatial-cluster）、动态演化（/dynamic-evolution）、省域诊断（/province-diagnosis）、类型识别（/type-identification）、SHAP解释（/shap）、布局决策（/layout-decision）、智能问答（/chat）。每个页面均为独立的客户端组件（"use client"），通过React Hooks（useState + useEffect）管理本地状态，使用原生fetch API调用后端数据接口，并在数据加载过程中展示LoadingSpinner加载动画，在出错时展示ErrorMessage错误提示（含重试按钮）。

前端封装了10个可复用的ECharts图表组件：DonutChart（环形饼图）、RoseChart（南丁格尔玫瑰图）、RankingBar（横向排名条）、ChinaMap（中国地图choropleth）、LineChart（折线图）、AreaChart（折线面积图）、StackedBar（堆叠柱状图）、BarChart（柱状图）、ShapWaterfall（SHAP瀑布图）、RegionChart（区域柱状图）。所有图表组件统一采用深色卡片容器、科技风配色方案和渐进式动画效果。

### 9.2.5 应用交互层

应用交互层是用户与系统交互的入口。用户通过浏览器访问系统URL（本地开发环境为http://localhost:3000，公网部署通过ngrok隧道暴露），经Next.js内置的API代理（/api/*路径自动转发至后端8000端口）与FastAPI后端通信。用户在首页获得全国宏观概览后，可依次浏览综合评价排名、区域差异分析、空间聚集模式、动态演化规律，也可通过省份选择器深入查看特定省份的诊断画像、SHAP解释和布局建议，或通过自然语言向智能问答模块提出个性化问题。

### 9.3.2 页面结构与布局设计

系统采用数据驾驶舱的标准三栏布局，由以下三个核心布局组件构成：

**（1）导航侧边栏（Sidebar）。** 左侧固定224px宽度的垂直导航栏，背景为深蓝灰色（#0f172a），按功能领域分为4个导航分组：

- **总览组**：首页总览（1项）
- **评价分析组**：综合评价、区域差异、空间聚焦、动态演化（4项）
- **诊断决策组**：省域诊断、类型识别、SHAP解释、布局决策（4项）
- **智能组**：智能问答（1项）

每个导航项使用Emoji图标+中文标签，当前激活页面以蓝色（#0ea5e9）高亮标识和左侧边框指示器突出显示。侧边栏底部展示系统版本信息"Green Compute v1.0 - 公开版"。

**（2）顶部状态栏（Header）。** 顶部固定56px高度的信息栏，居中展示系统全称"省域绿色算力承载能力评估与资源布局决策支持系统"，右侧展示数据状态标签"2016-2024"、"31 省份"和"数据就绪"，以绿色圆点标识数据完整性。

**（3）中央内容区。** 采用弹性布局填充剩余空间，垂直方向可滚动。每个页面使用PageContainer组件统一封装页面标题、副标题和内容区，内容区以响应式网格布局组织统计卡片（StatCard）和图表卡片。统计卡片以左侧彩色边框（蓝色/绿色/橙色/红色）区分信息类型，展示标签、数值和辅助说明。图表卡片统一使用深色背景（#1e293b）、圆角边框和1px灰色边框，内边距统一，视觉风格一致。

### 9.3.3 前端功能页面设计

系统共实现10个功能页面，每个页面的设计、数据来源和图表类型如下：

**（1）首页总览（/dashboard）。** 定位为系统的"数据驾驶舱入口"。页面顶部展示4个统计卡片（全国平均得分、最高省份、最低省份、区域数量），中央以大型环形饼图展示四大区域（东部、中部、西部、东北）的平均得分分布，右侧以横向排名棒棒糖图展示31省综合得分排名（前3名以金/银/铜三色突出标识）。数据来源于`GET /api/overview?year=2024`和`GET /api/scores?year=2024`两个接口。

**（2）综合评价（/evaluation）。** 以TOPSIS综合得分为核心，左侧展示南丁格尔玫瑰图（扇形半径映射区域平均得分，四色区分四大区域），右侧以横向排名条图展示31省得分并按从高到低排序，下方提供完整的数据排名表格（包含排名、省份、得分、年份列，前3名附奖牌图标）。数据来源于`GET /api/scores?year=2024`和`GET /api/overview?year=2024`。

**（3）区域差异（/regional-difference）。** 基于Dagum基尼系数分解结果，从四个维度展示区域差异特征：左上为折线面积图展示2016—2024年总体基尼系数变化趋势，右上为堆叠柱状图逐年分解区域内差异、区域间净差异和超变密度的贡献率变化，左下为横向排名条对比2024年六个区域组合之间的基尼系数高低，右下为柱状图展示四大区域内部的基尼系数差异。四幅图分别对应趋势分析、结构分解、组间比较和组内比较四个分析视角。数据来源于`GET /api/dagum`。

**（4）空间聚焦（/spatial-cluster）。** 以中国地图为页面核心可视化载体。地图采用choropleth（分级统计图）方式将31个省份的综合得分映射为连续渐变色（由浅到深的橙红色系），支持鼠标悬停显示省份名称、综合得分、全国排名和LPA类型。地图下方布置全局Moran's I指数的时间序列折线图（展示2016—2024年空间自相关程度的变化趋势）、LISA类型分布环形饼图（展示高-高、低-低、高-低、低-高四种聚集模式的占比）和LISA分析明细数据表（31省×得分×Z值×局部Moran's I×聚集类型）。数据来源于`GET /api/moran`、`GET /api/scores?year=2024`和`GET /api/lpa`。中国地图GeoJSON数据由前端`public/china.json`文件提供。

**（5）动态演化（/dynamic-evolution）。** 从时间维度展示省域绿色算力承载能力的动态变化规律。左侧以面积图展示四种水平状态（低水平、中低水平、中高水平、高水平）的转移概率时序变化，右侧以折线图展示空间Markov条件下向上跃迁的概率趋势（紫色#8b5cf6）。页面同时展示传统Markov转移概率矩阵（4×4热力图表格）和状态分类阈值说明面板。数据来源于`GET /api/markov`和`GET /api/scores?year=2024`。

**（6）省域诊断（/province-diagnosis）。** 为每个省份提供个性化的综合画像和短板诊断。页面初始默认展示上海数据（非空白状态），用户可通过省份选择器切换。顶部展示4个统计卡片（综合得分、全国排名、所属区域、LPA类型），中部以横向排名条图展示首要和次要障碍维度的障碍度，底部以结构化卡片展示布局建议（推荐布局类型、功能定位、优化策略、风险提示）。数据来源于`GET /api/provinces`、`GET /api/province/{province}/profile?year=2024`和`GET /api/obstacles?province=`。

### 9.3.5 前端数据流与状态管理

系统前端未引入Redux、Zustand等全局状态管理库，每个页面均为独立的React客户端组件，采用"数据获取—状态管理—渲染"的标准模式：

（1）**数据获取层**：所有后端API调用统一封装在`frontend/src/lib/api.ts`文件中，每个API函数对应一个后端接口，使用原生`fetch` API发起HTTP请求，返回类型安全的TypeScript接口对象。系统共封装了12个数据API函数：`fetchProvinces()`、`fetchScores(year)`、`fetchProvinceProfile(province, year)`、`fetchObstacles(province)`、`fetchLpa()`、`fetchShap(province, year)`、`fetchLayout(province)`、`fetchOverview(year)`、`fetchIndicators()`、`fetchDagum()`、`fetchMoran()`、`fetchMarkov()`。智能问答模块的API调用独立封装在`frontend/src/lib/api/chat.ts`中，实现了3次重试机制（指数退避1s、2s）、45秒超时控制和用户友好的错误信息转换。

（2）**状态管理层**：每个页面通过React的`useState` Hook管理四类状态：data（业务数据）、loading（加载状态）、error（错误信息）、selectedProvince（省份选择器状态）。`useEffect` Hook在页面挂载和省份切换时触发数据获取，自动管理loading/error/data的状态转换。

（3）**渲染层**：页面组件根据状态渲染对应的UI：loading=true时展示LoadingSpinner居中旋转动画；error非空时展示ErrorMessage错误卡片（含错误描述和重试按钮）；data非空时渲染统计卡片、图表组件和数据表格；data为空时展示EmptyState空状态占位提示。

## 9.4 后端服务设计与接口实现

### 9.4.1 后端技术框架

系统后端基于Python 3和FastAPI框架构建。FastAPI是基于Starlette和Pydantic的现代Python Web框架，具有以下关键特性：

- **异步高性能**：基于ASGI（Asynchronous Server Gateway Interface）标准，支持异步请求处理，适合I/O密集型的数据库查询和外部API调用场景。
- **自动API文档**：基于OpenAPI规范自动生成交互式Swagger UI文档（`/docs`端点）和ReDoc文档（`/redoc`端点），开发者可直接在浏览器中测试所有API接口。
- **Pydantic数据校验**：通过Pydantic v2模型自动校验请求参数类型、范围（如年份限定为2016—2030）和必填字段，无效请求在路由层即被拦截并返回422错误，避免无效查询传递到数据库层。
- **依赖注入**：通过FastAPI的Depends机制将数据库会话（`get_db()`）注入路由函数，实现数据库连接的自动获取和释放。

后端启动命令为：`uvicorn backend.app.main:app --host 0.0.0.0 --port 8000`（开发模式附加`--reload`热重载）。生产部署时通过`start_public.sh`脚本以崩溃自动重启模式运行（`while true`循环包裹uvicorn进程）。

### 9.4.4 数据校验与异常处理

系统通过多层校验机制确保数据的完整性和接口的健壮性：

**（1）Pydantic请求参数校验。** 路由函数的查询参数通过Pydantic类型注解实现自动校验。例如，年份参数通过`Query(2024, ge=2016, le=2030)`约束为2016—2030范围内的整数，超出范围时FastAPI自动返回HTTP 422（Unprocessable Entity）并附带详细的校验错误信息。省份名称参数通过`Query(...)`标记为必填（`...`是Pydantic的Required标识），缺失时自动返回422错误。

**（2）数据库查询异常映射。** 服务层函数在数据未找到时抛出`ValueError`（附带描述性错误信息），路由层捕获后转换为HTTP 404（Not Found）响应。例如，查询不存在的省份时，路由返回`404 - Province 'xxx' not found`并提供有效的省份名称列表。其他未预期的异常（数据库连接失败、SQL语法错误等）被捕获后转换为HTTP 500（Internal Server Error）响应，并记录详细的错误日志，但不在响应中暴露内部实现细节。

**（3）智能问答安全校验。** 智能问答接口（`POST /api/chat`）实现三层安全防护：第一层为问题长度限制（1—2000字符，由Pydantic的Field约束）；第二层为项目相关性检查（`_is_relevant()`函数匹配77个中文关键词，包括31个省份名称、领域术语和模型方法名），不相关的问题不调用LLM，而是直接返回友好的引导性回答（`category="out_of_scope"`），引导用户提出与绿色算力相关的问题；第三层为全链路异常捕获（`chat()`函数的`try-except`包裹整个处理流程），任何环节出错都返回友好的错误提示文本而非HTTP 500，确保前端始终能展示可读的回复而非"服务暂时不可用"。此外，系统配置了请求频率限制逻辑（`rate_limit.py`中的内存计数器，10次/60秒），但在当前公开版部署中因所有用户共享同一代理IP而暂时禁用。

| 异常类型 | HTTP状态码 | 触发场景 | 返回内容 | 前端处理 |
|----------|-----------|----------|----------|----------|
| Pydantic校验失败 | 422 | 参数类型错误、年份越界、必填字段缺失 | 详细的字段级错误信息 | 开发阶段控制台调试 |

### 9.4.5 跨域与前后端协作

系统采用前后端分离架构，开发环境中前端运行于`localhost:3000`，后端运行于`localhost:8000`。为解决跨域问题，系统在前后端各配置了一层代理：

- **后端CORS配置**：FastAPI应用通过`CORSMiddleware`中间件配置`allow_origins=["*"]`（开发阶段允许所有来源）、`allow_credentials=True`、`allow_methods=["*"]`、`allow_headers=["*"]`。在开发环境中，前端可直接跨域调用后端接口。
- **前端API代理**：Next.js通过`next.config.js`的`rewrites`配置将所有`/api/:path*`请求代理转发至`http://127.0.0.1:8000/api/:path*`。在生产部署模式下，前端静态资源和API请求统一由Next.js服务器处理，对外暴露单一端口（3000），后端8000端口仅内部访问，无需对外暴露。

这种双层代理设计使得系统在本地开发（前后端独立端口）、生产部署（单一端口+ngrok）和云端部署（Nginx反向代理）等不同场景下均可无缝工作。

## 9.5 数据库与数据组织设计

### 9.5.1 数据组织逻辑

系统以PostgreSQL关系型数据库（版本兼容12+）为核心数据存储引擎，通过SQLAlchemy 2.0 ORM框架实现Python代码与数据库表结构的映射。数据库设计遵循"年份—省份—指标—模型结果—页面展示"的主线组织逻辑：

- **年份维度**：所有时间序列数据（指标值、得分、基尼系数、Moran's I等）以`year`字段标识年度，支持2016—2024年的纵向查询和趋势分析，年份范围可通过更新数据表扩展至未来年度。

### 9.5.3 数据访问方式

后端通过SQLAlchemy ORM统一访问PostgreSQL数据库，数据访问遵循以下模式：

（1）**数据库连接管理。** FastAPI应用通过`backend/app/database.py`创建SQLAlchemy引擎（连接池pool_size=5, max_overflow=10, pool_pre_ping=True），并通过`get_db()`生成器函数为每个请求提供独立的数据库会话（Session），请求处理完成后自动关闭会话并归还连接到连接池。

（2）**ORM查询封装。** 每个服务模块通过SQLAlchemy的声明式查询API访问对应的模型表，例如：`db.query(TopsisScore).filter(TopsisScore.year == year).order_by(TopsisScore.rank).all()`。查询结果以ORM模型对象列表返回，路由层通过Pydantic的`model_validate()`方法将ORM对象转换为类型安全的响应模型（`from_attributes=True`配置启用ORM属性读取）。

（3）**跨表关联查询。** 省份综合画像接口（`GET /api/province/{province}/profile`）演示了跨表查询的典型模式：先分别查询`TopsisScore`、`LpaProvinceAssignment`、`ObstacleProvince`和`LayoutProvinceDecision`四张表（通过统一的`province`字段关联），再将四个查询结果合并填充到`ProvinceProfileOut`响应模型的14个字段中。

（4）**智能问答工具查询。** 智能问答模块的10个数据库工具函数（`backend/agent/tools.py`）使用独立的数据库引擎创建临时连接，直接执行查询并返回JSON序列化的结果字典。这种独立连接设计避免了与FastAPI的依赖注入会话冲突。

### 9.5.4 数据一致性与扩展性

系统在数据一致性和扩展性方面采取了以下设计策略：

- **数据导入校验**：Excel数据导入管道（`backend/importers/`）在数据入库前执行多层校验——必填字段非空检查、唯一键重复检查、年份范围验证（2016—2024）、概率值范围验证（[0, 1]）、省份名称标准化映射、区域归属验证等。
- **年度扩展支持**：所有含`year`字段的数据表均支持新增年度数据。在Excel源文件中添加下一年度数据后，重新运行数据导入脚本即可完成扩展，前端页面和API接口无需修改——折线图、面积图和排名表会自动展示新年度数据。
- **省份维度查询**：所有以省份为粒度的数据表和API接口均支持按单一省份查询，用户通过前端省份选择器切换即可获得不同省份的个性化分析结果。
- **模型结果更新**：当前文章节的模型方法使用新参数或新数据重新计算后，只需更新对应数据库表的内容，前端页面和API接口即可自动反映最新结果，无需重新部署系统。

## 9.6 智能问答模块设计与实现

### 9.6.1 功能定位与设计目标

智能问答模块是系统的人机交互入口，其设计目标是为非技术用户（政府部门决策者、企业规划人员）提供自然语言查询能力，降低系统使用门槛。用户不需要了解TOPSIS、Dagum、SHAP等模型的技术细节，也不需要知道数据存储在哪些数据库表中，只需以自然语言提出问题，系统即可自动完成问题理解、知识检索、数据库查询和回答生成的全流程。

模块的"智能"体现在三个方面：一是自动识别用户意图（排名查询、诊断分析、布局建议、概念解释或趋势预测），二是自动检索相关知识库文档为回答提供理论依据，三是自动调用数据库工具获取实时数据作为回答的事实基础。

### 9.6.2 技术架构与处理流程

智能问答模块的技术架构由四个核心组件构成：

**（1）问题路由器（Question Router）。** 负责对用户问题进行两级分类。首先通过关键词匹配（`_is_relevant()`）进行相关性判定：问题中若包含31个省份名称或"算力、绿色、TOPSIS、排名、障碍、布局"等46个领域术语中的任意一个，则判定为相关；否则直接返回`out_of_scope`引导性回答，不消耗LLM调用成本。相关性问题通过DeepSeek LLM进行分类（附带规则兜底），将问题归入6个类别：`concept_explanation`（概念解释）、`data_query`（数据查询）、`province_diagnosis`（省域诊断）、`layout_recommendation`（布局建议）、`future_prediction`（未来预测）、`unsupported`（暂不支持）。DeepSeek不可用时自动降级为基于关键词的规则分类。

**（2）RAG检索器（RAG Retriever）。** 基于scikit-learn的`TfidfVectorizer`实现文档检索（使用`char_wb`字符级分析器适配中文文本，max_features=2000, ngram_range=(1,2)）。知识库包含10篇Markdown文档，覆盖系统概述、TOPSIS模型说明、LPA模型说明、空间演化分析、障碍度模型、SHAP解释、布局决策规则、未来预测规则、指标体系参考和常见问题解答。检索器在应用启动时加载持久化的TF-IDF索引（`data/vector_store/rag_index.pkl`），通过余弦相似度计算返回与用户问题最相关的Top 5文档片段。检索到的文档片段被注入LLM的上下文，使回答具有可追溯的理论依据。

**（3）数据库工具集（Database Tools）。** 封装了10个数据库查询工具函数，覆盖系统的全部数据查询场景：`get_latest_scores()`查询最新年度所有省份得分、`get_province_score()`查询指定省份得分、`get_province_trend()`查询省份9年得分趋势及增长率、`get_top_rankings()`查询Top N排名、`get_lpa_result()`查询省份LPA类型、`get_obstacle_result()`查询省份障碍度诊断、`get_shap_top_features()`查询省份Top 8 SHAP特征贡献（含正负向分类）、`get_layout_recommendation()`查询省份布局决策详情、`compare_provinces()`多省份横向对比、`get_future_potential_ranking()`综合6因子计算省份未来潜力排名（公式为：0.30×当前得分 + 0.25×增长率 + 0.15×Markov跃迁概率 + 0.15×能源低碳优势 + 0.10×SHAP正向贡献 + 0.05×障碍度反向）。工具调度器根据问题类别自动选择调用哪些工具——例如"排名查询"触发`get_top_rankings()`，"省域诊断"触发`get_province_score()`、`get_lpa_result()`、`get_obstacle_result()`和`get_shap_top_features()`四个工具。

**（4）回答生成器（Answer Generator）。** 调用DeepSeek Chat API（模型：`deepseek-chat`，temperature=0.5，max_tokens=2048，超时60秒）生成最终的自然语言回答。系统提示词要求回答按照"结论→数据依据→模型依据→解释逻辑→建议→局限性说明"六段式结构组织，保证回答可追溯、可解释、可验证。DeepSeek API不可用时自动降级为离线模式，直接将数据库工具返回的JSON结果格式化后展示给用户，并附注"AI模型暂时不可用"的提示。

### 9.6.4 安全与稳定性设计

智能问答模块在安全与稳定性方面采取了多层防护措施：

- **问题长度限制**：前端输入框和Pydantic模型双重限制问题长度为1—2000字符，防止超长输入导致LLM处理异常。
- **相关性拦截**：非绿色算力领域的问题在关键词匹配阶段即被拦截，不消耗LLM API调用配额，并返回系统能力范围说明和推荐问题示例（`category="out_of_scope"`）。
- **LLM降级策略**：DeepSeek API不可用时，系统自动降级为基于数据库工具查询结果的离线回答模式，保证核心数据查询功能始终可用。
- **全链路兜底**：`chat()`函数外层被`try-except`包裹，任何未预期的异常都不会导致HTTP 500错误，而是返回友好的中文错误提示。
- **请求频率限制**：系统在代码层面预留了基于IP的请求频率限制逻辑（10次/60秒），在需要时可通过配置启用。
- **超时控制**：前端聊天API客户端设置了45秒的请求超时（AbortController），后端DeepSeek API调用设置了60秒的超时，平台级超时配置的后续可进一步完善。

### 9.7.2 构建与启动方式

系统通过项目根目录的`start_public.sh`脚本实现一键启动，脚本按顺序执行以下四个步骤：

**（1）后端启动。** 以崩溃自动重启模式运行FastAPI应用：`python3 -m uvicorn backend.app.main:app --host 127.0.0.1 --port 8000 --log-level warning`。采用`while true`循环包裹uvicorn进程，确保后端崩溃后3秒内自动重启，适用于比赛展示和演示场景。

**（2）前端构建。** 进入`frontend/`目录，生成运行时环境配置（`NEXT_PUBLIC_API_BASE_URL=`和`NEXT_PUBLIC_USE_MOCK=false`），执行`npx next build`生成生产构建产物。构建过程包括TypeScript类型检查、Tailwind CSS摇树优化和Next.js静态页面/服务端渲染打包。

**（3）前端启动。** 以生产模式启动Next.js服务器：`npx next start --port 3000 --hostname 0.0.0.0`。Next.js的`output: "standalone"`配置确保生产构建产物包含运行所需的最小Node.js运行时，无需完整安装`node_modules`。

**（4）公网隧道。** 使用ngrok创建HTTP隧道将本地3000端口暴露为公网HTTPS URL：`ngrok http 3000`。脚本自动从ngrok本地API（`127.0.0.1:4040/api/tunnels`）获取公网访问地址并在终端输出，用户可通过该地址在任何联网设备上访问系统。

### 9.7.3 环境变量与配置管理

系统的环境变量通过两个`.env`文件分层管理：

- 项目根目录`.env`：存储后端必需的敏感配置（`DATABASE_URL`、`DEEPSEEK_API_KEY`），由Python的`python-dotenv`库加载。
- `frontend/.env.local`：存储前端运行时配置（`NEXT_PUBLIC_API_BASE_URL`、`NEXT_PUBLIC_USE_MOCK`），由Next.js构建时自动注入。变量以`NEXT_PUBLIC_`前缀命名以暴露给浏览器端JavaScript。

`.env.example`文件提供环境变量模板，不含实际密钥值。部署时需手动创建`.env`文件并填入真实的数据库连接字符串和API密钥。系统通过`.gitignore`确保`.env`文件不被提交到版本控制系统。

### 9.7.4 系统运行检查

系统部署完成后，可通过以下步骤进行运行状态检查：

1. **后端健康检查**：访问`http://127.0.0.1:8000/health`，应返回`{"status":"ok","service":"green-compute-api"}`。
2. **API接口检查**：访问`http://127.0.0.1:8000/docs`查看Swagger UI交互式API文档，逐一测试12个数据接口是否正常返回JSON数据。
3. **前端页面检查**：访问`http://localhost:3000/dashboard`，确认首页总览正常加载统计卡片和图表。
4. **智能问答检查**：
   ```bash
   curl -X POST http://127.0.0.1:8000/api/chat \
     -H "Content-Type: application/json" \
     -d '{"question":"上海绿色算力排名是多少"}'
   ```
   应返回包含上海排名信息、category为`data_query`或`province_diagnosis`的JSON响应。
5. **无关问题测试**：
   ```bash
   curl -X POST http://127.0.0.1:8000/api/chat \
     -H "Content-Type: application/json" \
     -d '{"question":"你好"}'
   ```
   应返回category为`out_of_scope`的引导性回答（非500错误），引导用户提出绿色算力相关问题。
6. **数据库连接检查**：确认PostgreSQL服务正在运行，`green_compute`数据库可正常连接。

## 9.8 系统功能展示与应用价值

### 9.8.1 面向政府部门的宏观监测

系统为发改、工信、能源等政府部门提供全国绿色算力承载能力的全景视图和动态监测工具。通过首页总览页面，决策者可在一屏内获取全国平均得分、最优/最弱省份、区域得分分布和31省完整排名等宏观指标。通过区域差异分析页面，政策制定者可精确量化东部、中部、西部和东北四大区域之间差异的来源结构（区域内差异、区域间净差异、超变密度），为"东数西算"工程的空间布局优化和转移支付政策制定提供数据支撑。空间聚焦页面基于中国地图的交互式可视化，使决策者能够直观识别绿色算力承载能力的空间聚集模式和冷热区域，支撑跨区域协同治理决策。

### 9.8.2 面向数据中心企业的选址决策

数据中心企业可通过系统的布局决策模块，基于综合适宜度指数、能源低碳优势指数、需求网络优势指数和约束压力指数等多个量化指标，科学评估31个省份的数据中心选址适宜性。系统为每个省份提供了推荐布局类型（需求驱动型、能源驱动型、均衡发展型、基础培育型）、功能定位、优化策略和风险提示，企业规划人员可结合自身业务需求（如靠近用户市场、降低电力成本、获取绿色能源认证）做出数据驱动的选址决策。未来潜力预测功能为企业的5—10年长期战略规划提供趋势参考。

### 9.8.3 面向能源规划部门的低碳约束分析

能源规划部门可通过系统综合评价和障碍度诊断功能，了解各省在可再生能源利用、碳排放控制和能源效率方面的表现。SHAP解释模块揭示了绿色低碳约束维度各指标对综合得分的贡献方向和大小，帮助规划部门识别哪些低碳指标对各类型省份的影响最为显著。布局决策模块中的能源低碳优势指数和约束压力指数，可直接服务于"双碳"目标下的绿色算力资源配置决策。

### 9.8.4 面向区域协调发展的差异治理

系统通过Dagum基尼系数分解、Moran's I空间自相关分析和LPA类型识别，揭示了四大区域之间和区域内部的发展差异来源和演变趋势。动态演化页面基于Markov转移矩阵展示了省份在不同发展水平之间的转移概率和演变路径，为识别"上升潜力省份"和"下降风险省份"提供量化依据。类型识别页面将31个省份划分为四种发展类型，每类给出差异化的提升策略，支撑精准化的区域协调发展政策制定。

### 9.8.5 面向非技术用户的智能问答入口

智能问答模块是系统降低使用门槛的核心设计。非技术用户不需要理解TOPSIS、SHAP、Dagum等模型原理，也不需要学习SQL查询或数据分析工具，只需以自然语言输入问题（如"贵州为什么排名靠后？""哪些省份适合建数据中心？""什么是障碍度？"），系统即可在秒级内返回结构化且附有数据依据和模型来源的回答。回答中的"数据依据"和"模型依据"提供了事实来源追溯，用户可通过数据源卡片（Data Source Card）查看系统调用了哪些数据库工具和检索了哪些知识文档，保证了回答的透明性和可信度。

### 9.8.6 系统相比静态分析报告的核心优势

本系统将前文章节的静态模型分析结果转化为可交互、可更新、可查询、可解释、可扩展的数字化决策支持平台，相比传统论文中的静态图表和离线分析报告，具有以下核心优势：

- **可交互**：所有图表支持hover、zoom、筛选、切换等交互操作，用户可自主探索数据，而非被动接收固定结论。
- **可更新**：新增年度数据后重新运行导入脚本即可完成全系统数据更新，前后端无需代码修改。
- **可查询**：用户通过省份选择器或自然语言即可获取特定省份的个性化分析结果，而非一篇报告中只能展示少数典型案例。
- **可解释**：SHAP解释和障碍度诊断模块为每个结论提供了模型级和特征级的可解释性支持。
- **可扩展**：系统的五层架构设计使每一层均可独立扩展——可接入新的模型分析结果、增加新的数据表、添加新的前端页面或扩展智能问答的工具集。
- **可部署**：系统以B/S架构和容器化构建产物交付，任何联网设备均可通过浏览器访问，无需安装专用软件。

## 9.9 本章小结

本章系统介绍了"省域绿色算力承载能力评估与资源布局决策支持系统"的设计理念、技术架构、功能实现和部署方案。系统以前文章节的模型分析成果为基础，将其工程化封装为可交互、可查询、可展示的Web数据驾驶舱，并通过RAG检索增强生成技术实现了自然语言智能问答，构建了从"数据汇聚—模型评价—空间诊断—类型识别—布局决策—智能问答"的完整应用闭环。

在技术架构方面，系统采用前后端分离的B/S架构：后端基于FastAPI + SQLAlchemy + PostgreSQL技术栈，提供12个RESTful数据API接口和1个智能问答接口，通过分层架构（路由层→服务层→模型层）实现了业务逻辑的清晰分离和代码复用；前端基于Next.js 15 + React 19 + TypeScript + Tailwind CSS + ECharts 5.5技术栈，实现了10个功能页面的数据驾驶舱，封装了10个可复用的ECharts图表组件，并通过Next.js API代理实现了前后端的无缝协作。

在数据组织方面，系统以PostgreSQL数据库为核心，通过42张数据表统一管理原始指标数据、模型计算结果和知识库文档，以"年份—省份—指标—模型结果"为主线组织数据关系，支持按年度扩展、按省份筛选和跨模型联合查询。

在智能问答方面，系统集成DeepSeek大语言模型，通过"问题分类→RAG检索→数据库工具调用→LLM生成回答"的四步处理流程，实现了对6类问题的自动理解和结构化回答。系统配置了多层安全防护机制（关键词相关性拦截、LLM降级策略、全链路异常兜底），确保智能问答服务的稳定性和安全性。

在部署方面，系统通过`start_public.sh`脚本实现一键启动（后端崩溃自恢复 + 前端生产构建 + ngrok公网隧道），支持本地开发和公网展示两种运行模式。

系统将前章节的模型分析成果从"静态论文结论"转化为"动态决策服务"，体现了数据要素从"数据资源"到"分析结果"再到"决策服务"的价值转化路径，为政府部门、数据中心企业、能源规划机构和研究人员提供了一站式的绿色算力承载能力评估与资源布局决策支持工具。系统当前版本为公开版（v1.0），后续可进一步完善的功能包括：用户认证与权限管理、地级市数据粒度扩展、实时数据流接入、多模型对比分析、报告自动生成和移动端适配等。
